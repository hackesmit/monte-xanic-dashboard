1. RESUMEN GENERAL
================================================================================

El Dashboard Monte Xanic es una aplicación web de analítica vinícola diseñada
para el seguimiento de maduración y fenólicos de la bodega Monte Xanic en
Valle de Guadalupe, Baja California. Rastrea la química de bayas (Brix, pH,
tANT, AT), fenólicos del vino, tasas de extracción y comparaciones entre
vendimias para más de 13 variedades de uva tinta, 4 blancas y 10+ orígenes
de viñedo en tres valles (VDG, VON, SV).

Fases completadas:
  - Fase 1: Despliegue en Vercel
  - Fase 2: Migración a base de datos Supabase
  - Fase 3: Integración meteorológica (Open-Meteo)
  - Fase 4: Autenticación (bcrypt + HMAC, 2h expiración)
  - Fase 4b: Remodelación de datos y visualización
  - Fase 4c: Estabilidad, seguridad y mejoras de visualización
  - Fase 5: Mapa de calidad de viñedos (SVG)
  - Fase 6: Pulido completo (login, PDF, filtros móviles, tendencias,
            radar, calendario de cosecha)
  - Endurecimiento de seguridad (upload servidor, rate limit persistente,
    revocación de tokens, endpoint de logout)
  - Ronda 4 de revisión (helper compartido de extracción, correcciones
    menores de código)

Fase pendiente:
  - Fase 7: Mediciones Técnicas con Evidencia Fotográfica (diferida)

URL de producción: https://monte-xanic-dashboard-ky5t.vercel.app/
Escala de datos: ~3,500 muestras, ~500–800 nuevas por vendimia

3.19 tANT: Baya vs Vino (chartExtraction)
     Tipo: Barras horizontales agrupadas
     Función: createExtractionChart()
     Descripción: Comparación directa de tANT de baya vs tANT de vino

3.20 Tasa de Extracción (%) (chartExtractionPct)
     Tipo: Barras horizontales
     Función: createExtractionPctChart()

     descendente. Codificado por bandas de calidad:
       - Verde (>50%): buena extracción
       - Dorado (30–50%): extracción media
       - Rojo (<30%): extracción baja
     Tooltip muestra: varietal, origen, tANT baya, tANT vino, %.
     Eje X sin límite máximo fijo — valores >100% se muestran
     completamente (no recortados).
     Usa el mismo helper _buildExtractionPairs() que el gráfico 3.19.
     Ejes: Y = lote, X = extracción (%)

3.25 Calendario de Cosecha (chartHarvestCal) ★ NUEVO
     Tipo: Mixto (barras flotantes + línea + barras de lluvia)
     Función: createHarvestCalendar()
     Altura: 360px
     Descripción: Línea de tiempo horizontal mostrando la ventana de
     cosecha por varietal, superpuesta con datos climáticos diarios.

     Componentes del gráfico:
       a) Barras de cosecha — una barra horizontal flotante por varietal,
          desde la primera hasta la última fecha de cosecha (crushDate)
          de ese varietal en la vendimia seleccionada. Coloreada con el
          color de la variedad (CONFIG.varietyColors).

       b) Línea de temperatura — temperatura media diaria (°C) como
          línea semi-transparente sobre eje Y secundario. Datos del
          valle más relevante según los datos filtrados (VDG por defecto).

       c) Barras de lluvia — precipitación diaria (mm) como barras
          verticales delgadas detrás de las barras de cosecha, en eje
          Y secundario.

     Tooltip: varietal, rango de fechas de cosecha, número de lotes
     cosechados, Brix promedio y tANT promedio al momento de cosecha.

     Eje X: fechas del calendario (Jul–Oct) con escala de tiempo
            (requiere adaptador chartjs-adapter-date-fns).
     Eje Y izquierdo: nombres de varietal.
     Eje Y derecho: temperatura (°C) / precipitación (mm).

     Comportamiento con múltiples vendimias seleccionadas: utiliza la
     vendimia más reciente para el calendario.

     Estado vacío: "Sin fechas de cosecha para esta vendimia"

     Ubicación en UI: entre los gráficos de comparación y la sección
     de clima. Sección "Calendario de Cosecha" con botón de exportación.

3.26 Temperatura Media Diaria (chartWeatherTemp)
     Tipo: Líneas múltiples
     Función: createWeatherTimeSeries()
     Descripción: Temperatura media diaria (°C) durante la temporada de
     cosecha (1 Jul – 31 Oct), con una línea por vendimia. Día 1 = 1 de
     julio. Permite comparar patrones térmicos entre años.
     Ejes: X = día de temporada (1–123), Y = °C

3.27 Eventos de Lluvia (chartWeatherRain)

     Ejes: X = día de temporada (1–123), Y = mm

5. TABLAS DE DATOS
================================================================================

5.1  Tabla de Muestras de Baya (vista Bayas — máx. 300 filas)
     Columnas: ID Muestra, Fecha, Vendimia, Varietal, Origen, Brix, pH,
     AT, tANT, Peso Baya, DPE (días post-envero)
     Características:
       - Ordenamiento ascendente/descendente por cualquier columna
       - Colores de alerta: pH >4.5 rojo, >3.9 alerta
       - Colores de alerta: Brix ≥24 rojo, ≥21 verde, <21 dorado
       - Marcador † para valores bajo límite de detección (<50, <10)
       - Escape XSS en todas las celdas
       - Varietal y Origen como badges con color

5.2  Tabla de Recepción de Tanque (vista Vino)
     Columnas: Código Bodega, Fecha, Tanque, Variedad, Origen, Tipo
     de Muestra, tANT (WX), fANT, pTAN, IPT, DPE
     Fuente: wine_samples con sample_type ≠ Berry

5.3  Tabla de Mostos / Pre-fermentativos (vista Vino)
     Columnas: Código Bodega, Fecha, Tanque, Variedad, Origen, tANT,
     fANT, pTAN, IPT, DPE
     Fuente: tabla prefermentativos + registros Must de wine_samples

5.4  Tabla de Detalle de Extracción (vista Extracción)
     Columnas: Lote Baya, Lote Vino, Varietal, Origen, tANT Baya,
     tANT Vino, Extracción %
     Fuente: mapeo berryToWine de config.js + helper _buildExtractionPairs

5.5  Tabla Resumen Comparativo (vista Vendimias)
     Columnas: Parámetro, [Vendimia A], [Vendimia B], Diferencia, Cambio %
     Muestra: Diferencias año a año en Brix, pH, tANT, AT promedios
     Se genera dinámicamente según las vendimias seleccionadas

5.6  Tabla Promedios por Varietal por Vendimia (vista Vendimias)
     Columnas: Varietal + columnas dinámicas por vendimia
     Muestra: Brix, pH, tANT, AT y conteo (n) por varietal × vendimia

================================================================================
6. SISTEMA DE FILTROS
================================================================================

El dashboard tiene dos conjuntos independientes de filtros: uno para datos
de baya (vistas Bayas, Extracción, Vendimias, Mapa) y otro para datos de
vino (vista Vino).

─── Filtros de Baya ───

  Vendimia:     Chips de selección múltiple (2022, 2023, 2024, 2025...)
  Tipo de Uva:  Radio (Todas / Tintas / Blancas)
  Varietal:     Chips de selección múltiple, coloreados por tipo tinta/blanca
  Origen:       Chips de selección múltiple, coloreados por rancho
  Lote:         Campo de búsqueda por texto + chips de lotes específicos
  Colorear por: Tabs (Por Varietal / Por Origen) — cambia el esquema de
                colores en todos los gráficos de dispersión y barras

─── Filtros de Vino ───

  Vendimia:     Chips de selección múltiple
  Tipo de Uva:  Radio (Todas / Tintas / Blancas)
  Varietal:     Chips de selección múltiple
  Origen:       Chips de selección múltiple

─── Funcionalidades de Filtrado ───

  - Todos los gráficos se actualizan en tiempo real al cambiar filtros
  - Los filtros se sincronizan visualmente al cambiar de vista
  - Los lotes obsoletos se eliminan automáticamente del estado de filtros
  - El botón "Limpiar Todo" reinicia todos los filtros a su estado default
  - Refresh con debounce para evitar actualizaciones excesivas
  - Guard de concurrencia (_refreshInProgress) para evitar renders paralelos
  - En móvil: panel de filtros como bottom sheet deslizable con:
    · Esquinas redondeadas (14px)
    · Barra de arrastre (pull handle)
    · Header con título "Filtros" y botón de cierre (×)
    · Animación slide-up / slide-down
    · Bloqueo de scroll del body durante uso
    · Secciones colapsables (Varietal/Origen, Evolución/Clima)

  - Resumen de filtros activos visible en modo móvil

7. FUNCIONALIDAD DE CARGA DE DATOS
================================================================================

El dashboard soporta carga de datos por arrastrar y soltar (drag & drop)
o selección de archivo. Solo disponible para usuarios con rol "lab".
Desde el endurecimiento de seguridad, todas las inserciones pasan por
el endpoint servidor /api/upload (ya no se insertan directamente desde
el cliente con la clave anon de Supabase).

─── Tipos de Archivo Soportados ───

7.1  CSV de WineXRay
     Formato: CSV exportado del software WineXRay
     Contenido: ID muestra, tipo de muestra, vendimia, variedad, origen,
     compuestos fenólicos (tANT, fANT, bANT, pTAN, iRPs, IPT), pH, AT,
     Brix, alcohol, A.V., ácido málico, azúcares residuales, L*/a*/b*,
     peso de baya, antocianinas de baya, azúcares de baya
     Destino: tabla wine_samples en Supabase
     Manejo especial:
       - Valores "<50", "<10" → NULL con flag below_detection = true
       - Valores ">100" → se detectan como above_detection
       - Filtrado automático de muestras de calibración de laboratorio:
         COLORPRO, CRUSH, WATER, BLUEBERRY, RASPBERRY, BLACKBERRY
       - Exclusión de muestras experimentales (EXP/EXPERIMENTO) y
         California
       - Normalización de variedad: Petite Sirah → Durif
       - Normalización de origen: formato viejo → rancho-primero
         (ej: "VDG - Monte Xanic" → "Monte Xanic (VDG)")

7.2  Excel de Recepción de Tanque
     Formato: .xlsx con dos hojas
     Hoja 1 "Recepción 2025": datos de recepción de tanque con hasta 4
       lotes de viñedo por fila
       → tabla tank_receptions + reception_lots (1 fila padre + N hijos)
     Hoja 2 "Prefermentativos 2025": datos pre-fermentativos
       → tabla prefermentativos
     Validación de vintage: prefijo de 2 dígitos del código de lote
       convertido a año (25 → 2025), rango válido 2015–2040

─── Pipeline de Carga ───

  1. Usuario arrastra archivo al dashboard (o selecciona manualmente)
  2. Detección automática de tipo (.csv → WineXRay, .xlsx → Recepción)
  3. Parseo del archivo en el cliente (SheetJS para Excel, nativo para CSV)
  4. Detección de duplicados (consulta a Supabase para composite keys)
  5. Envío al servidor: POST /api/upload con token de sesión
  6. Validación servidor: verifica token, rol, tabla, y límite de filas
  7. Upsert en Supabase con service key (lotes de hasta 500 filas)
  8. Confirmación en español: "✓ X muestras agregadas correctamente"
  9. Recarga automática de datos en el dashboard

─── Límites ───

  - Tamaño máximo de archivo: 10 MB
  - Filas por tabla: wine_samples 500, tank_receptions 200,
    reception_lots 2000, prefermentativos 200
  - Solo roles "lab" o "admin" pueden cargar datos

─── Función de Limpieza ───

  cleanupLabSamples(): elimina muestras de laboratorio (COLORPRO%,
  CRUSH%, WATER%, *BLUEBERRY*, etc.), muestras 'NORMAL', California,
  e IDs explícitamente excluidos.

8. MÓDULO METEOROLÓGICO
================================================================================

Integración con Open-Meteo Archive API para datos climáticos históricos
de tres valles vinícolas. Datos almacenados en Supabase para caché
persistente.

─── Valles Monitoreados ───

  VDG — Valle de Guadalupe   (32.08°N, -116.62°W)
  VON — Valle de Ojos Negros (32.00°N, -116.25°W)
  SV  — San Vicente           (32.05°N, -116.45°W)

─── Variables Meteorológicas por Día ───

  - Temperatura máxima diaria (°C)
  - Temperatura mínima diaria (°C)
  - Temperatura media diaria (°C)
  - Precipitación (mm)
  - Humedad relativa (%)
  - Índice UV
  - Velocidad del viento (km/h)

─── Funciones Calculadas ───

  Grados Día de Crecimiento (GDD):
    Fórmula: Σ max(0, temp_avg - 10) desde 1 Jul hasta fecha
    Base: 10°C
    Caché: resultados se almacenan para evitar recálculos
    Validación: retorna null si faltan >3 días o >10% del período

  Lluvia Acumulada:
    Suma de precipitación desde 1 Jul hasta fecha de muestreo
    Se usa para correlación con tANT

  Resolución por Valle:
    Cada muestra de baya se asocia al valle correcto según su
    origen/appellation (ej: Kompali → VON, Monte Xanic → VDG,
    Dubacano → SV). Fallback: VDG.

─── Sincronización y Caché ───

  - Tabla meteorology en Supabase con constraint único (date, location)
  - sync() detecta días faltantes y solo descarga los que no existen
  - Temporada de datos: 1 Jul – 31 Oct de cada vendimia activa
  - Guard de concurrencia: evita sincronizaciones paralelas
  - Validación de respuesta API: verifica que los arrays tengan datos

10. AUTENTICACIÓN Y SEGURIDAD
================================================================================

─── Sistema de Login ───

  - Pantalla de login con diseño estilizado:
    · Fondo con brillo dorado radial
    · Tarjeta con sombras escalonadas (layered box-shadow)
    · Logo con drop-shadow
    · Divisor con gradiente dorado
    · Campos con glow dorado en focus
    · Botón con gradiente dorado y hover glow
    · Animación de entrada secuencial (card → logo → divider →
      tagline → fields → button → footer)
  - Hash bcrypt para contraseñas (server-side, bcryptjs)
  - Tokens de sesión HMAC-SHA256 con expiración de 2 horas
  - Nonce aleatorio de 16 bytes en cada token (previene reutilización)
  - Almacenamiento local: token y rol en localStorage

─── Rate Limiting ───

  Persistente en Supabase (tabla rate_limits):
    - 10 intentos por IP en ventana de 15 minutos
    - Tabla: ip TEXT PRIMARY KEY, attempts INT, window_start TIMESTAMPTZ
    - Índice en window_start para limpieza eficiente
    - Fallback: rate limiting en memoria si Supabase no disponible
    - Extracción de IP: x-forwarded-for → x-real-ip → 'unknown'
    - Delay artificial de 300ms en fallos de autenticación (anti-timing)

─── Revocación de Tokens ───

  - Tabla token_blacklist: token_hash TEXT PRIMARY KEY, invalidated_at
  - Al hacer logout: se hashea el token con SHA256 y se inserta
  - Verificación: /api/verify consulta la blacklist antes de validar
  - Fail-open: si la DB no está disponible, el token sigue válido
    (prioriza disponibilidad sobre revocación)
  - Tokens expirados (>2h) se auto-invalidan sin necesidad de blacklist

─── Roles ───

  viewer: Acceso de solo lectura (predeterminado por seguridad en TODOS
          los fallbacks de error)
  lab:    Lectura + capacidad de carga de datos

─── Endpoints de API (Vercel Serverless) ───

  POST /api/login    — Autenticación con bcrypt
                       Entrada: { username, password }
                       Salida:  { ok, token } o { error }
                       Rate limit: 10/15min por IP

  POST /api/verify   — Validación de token + blacklist check
                       Entrada: { token }
                       Salida:  { valid, role }

  POST /api/logout   — Revocación de token (blacklist)
                       Entrada: { token }
                       Salida:  { ok: true } (siempre éxito para UX)

  POST /api/upload   — Carga de datos validada por servidor
                       Header:  X-Session-Token
                       Entrada: { table, rows, conflict? }
                       Validación: token válido + rol lab/admin +
                                   tabla permitida + límite de filas
                       Salida:  { ok, count } o { error }

  POST /api/config   — Credenciales de Supabase (protegido por auth)
                       Solo retorna SUPABASE_URL y ANON_KEY al cliente

─── Medidas de Seguridad Implementadas ───

  - Server-side upload: las inserciones a Supabase usan service key
    desde el servidor, no la anon key desde el cliente
  - Token blacklist verificada en /api/upload además de /api/verify
  - Escape XSS en todas las tablas (función _esc())
  - Headers CSP configurados en vercel.json
  - Headers de seguridad: Cache-Control: no-store, X-Content-Type-Options
  - Sin credenciales en código fuente (.env.local excluido de git)
  - Fallback de rol a 'viewer' (no 'admin') en todos los casos de error
  - Comparación timing-safe de nombres de usuario
  - bcrypt siempre verifica todas las cuentas (evita enumeración de
    usuarios por timing)
  - Guard de concurrencia en refresh() y weather sync
  - Longitud máxima de credenciales: username 128, password 1024
